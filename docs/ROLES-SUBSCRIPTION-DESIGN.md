# تصميم الصلاحيات والاشتراكات والشركات — BHD-OM

## المراجع
- نظام الاشتراك في **ain-oman-web**: خطط، حدود، صلاحيات، FeatureGate؛ سنطبّق أفضل منه (قاعدة بيانات واحدة، واجهات واضحة، تبعية عقار/جهة اتصال).
- **قواعد الظهور والإجراءات** (أ ب ج د ر): انظر `docs/ROLES-VISIBILITY-RULES.md` — من يرى ماذا، من يعتمد العقد، من متفرج/دعم فني.

---

## 1. الأدمن والصلاحيات

### 1.1 الأدمن الكامل (Super Admin)
- صلاحيات كاملة: كل البيانات، حذف، أرشفة، إعدادات الموقع.
- إنشاء حسابات أدمن جديدة (كاملة أو جزئية).

### 1.2 أدمن بصلاحيات جزئية
- الأدمن الكامل يحدد صلاحيات كل حساب أدمن، مثلاً:
  - إدارة المستخدمين فقط
  - إدارة العقارات فقط
  - المحاسبة فقط
  - التقارير والتحليلات فقط
  - أو مجموعة منها (مصفوفة صلاحيات).
- في الواجهة: عرض كل البيانات المسموح بها + عمود «تابع لـ» حيث يناسب.

### 1.3 نموذج البيانات (مقترح)
- **AdminRole** (أو صلاحيات مخزنة في User): قائمة صلاحيات (مثل `MANAGE_USERS`, `MANAGE_PROPERTIES`, `MANAGE_ACCOUNTING`, `FULL_ACCESS`, …).
- المستخدمون من نوع ADMIN يربطون بدور (أو قائمة صلاحيات). الأدمن الكامل: دور «كامل» أو علم `isSuperAdmin`.

## 2. الشركات وحسابات الموظفين

### 2.1 الشركة (Organization / Company)
- كيان شركة (اسم، بيانات، إلخ).
- الشركة لها **عقارات** و**جهات اتصال** و**حجوزات** مرتبطة بها.

### 2.2 حسابات الموظفين
- الشركة تنشئ حسابات موظفين (من لوحة تحكم الشركة).
- الموظف يرى فقط بيانات **شركته**: عقارات الشركة، جهات الاتصال، الحجوزات المرتبطة.
- الأدمن يرى كل شيء مع عرض «الشركة: X» و«تابع لـ المستخدم Y».

### 2.3 نموذج البيانات (مقترح)
- **Organization**: id, nameAr, nameEn, … مع ربطها بـ User (صاحب الشركة) وربط العقارات/جهات الاتصال بالشركة أو بالمستخدم المالك.
- **User**: حقل اختياري `organizationId` و/أو `ownedOrganizationId`؛ دور مثل `COMPANY_EMPLOYEE` أو استخدام نفس `OWNER` مع ربط تنظيمي.

## 3. نظام الاشتراك والباقات

### 3.1 الخطط (Plans)
- مخزنة في PostgreSQL (وليس ملفات أو localStorage).
- كل خطة: اسم، سعر، مدة (شهري/سنوي)، **ميزات** (قائمة أو خريطة صلاحيات)، **حدود** (عدد عقارات، عدد مستخدمين فرعيين، تخزين، إلخ).
- أمثلة: أساسي، قياسي، احترافي، شركات (مثل ain-oman-web لكن في DB).

### 3.2 اشتراك المستخدم (User Subscription)
- كل مستخدم (أو كل شركة) له اشتراك فعال: خطة، تاريخ بداية/نهاية، حالة (نشط/منتهي/ملغي).
- التحقق من الحدود: عند إضافة عقار أو مستخدم فرعي، التحقق من عدم تجاوز حد الخطة.

### 3.3 فتح/قفل الميزات
- واجهة واحدة (مثل SubscriptionContext + FeatureGate) تتحقق من:
  - هل المستخدم لديه اشتراك نشط؟
  - هل الخطة تتضمن الميزة المطلوبة؟
  - هل ضمن الحد (عدد العقارات، إلخ)؟
- في الواجهة: إخفاء أو تعطيل الأزرار/الصفحات غير المسموح بها مع رسالة «ترقية الاشتراك» أو «الاشتراك مطلوب».

### 3.4 تحسين عن ain-oman-web
- مصدر واحد للحقيقة: PostgreSQL (Plans, Subscriptions, Usage).
- واجهات API واضحة: `/api/plans`, `/api/subscriptions`, `/api/subscriptions/check`.
- عدم الاعتماد على localStorage للخطط أو حالة الاشتراك في الإنتاج.

## 4. عزل البيانات وعرض «تابع لـ»

### 4.1 القواعد
- **مستخدم عادي (عميل/مالك/موظف شركة)**:
  - يرى فقط العقارات/جهات الاتصال/الحجوزات **التابعة له أو لشركته**.
- **الأدمن (كامل أو جزئي)**:
  - يرى كل ما تسمح به صلاحياته.
  - في القوائم: عمود «تابع لـ» (المستخدم أو الشركة) بجانب العقار وجهة الاتصال.

### 4.2 تغييرات المخطط المقترحة
- **Property**: إضافة `createdById` أو `ownerId` (مرتبط بـ User) و/أو `organizationId` (مرتبط بـ Organization). الأدمن يرى الكل؛ غير الأدمن: فلترة حسب المستخدم/الشركة.
- **جهات الاتصال**: إنشاء نموذج **Contact** في DB (إن لم يوجد) مع `userId` أو `organizationId`؛ أو ربط ContactSubmission بمستخدم/شركة.
- **الحجوزات**: ربط أوضح بالمستخدم/الشركة إن لزم.

## 5. الأرشفة

### 5.1 مبدأ
- عناصر قابلة للأرشفة: عقار، جهة اتصال، مستخدم (تعطيل بدل حذف)، إلخ.
- حقل **isArchived** (وتاريخ أرشفة وسبب اختياري).

### 5.2 السلوك
- في واجهة المستخدم العادي: عدم إظهار العناصر المؤرشفة (أو إظهارها في تبويب «أرشيف»).
- في لوحة الأدمن: عرض الكل مع فلتر «مؤرشف / غير مؤرشف» وإمكانية استرجاع من الأرشيف.

---

## 6. ترتيب التنفيذ المقترح

1. **المرحلة 1 — الصلاحيات والأدمن**
   - توسيع User/Admin (أدمن كامل vs جزئي، مصفوفة صلاحيات).
   - واجهة إدارة صلاحيات الأدمن وإنشاء حسابات أدمن جديدة.

2. **المرحلة 2 — عزل البيانات و«تابع لـ»**
   - إضافة حقول المالك/الشركة للعقارات وجهات الاتصال في DB.
   - تطبيق الفلترة في APIs وواجهة الأدمن (عمود «تابع لـ»).

3. **المرحلة 3 — الشركات والموظفين**
   - نموذج Organization وحسابات الموظفين وربط البيانات بالشركة.

4. **المرحلة 4 — الاشتراكات والباقات**
   - نماذج Plan و Subscription وربطها بالمستخدم/الشركة.
   - واجهة اختيار الخطة والتحقق من الحدود والميزات (أفضل من ain-oman-web).

5. **المرحلة 5 — الأرشفة**
   - حقول أرشفة واسترجاع وواجهة أدمن للأرشيف.

## تطبيق التغييرات على القاعدة

بعد سحب التحديثات التي تضيف الصلاحيات والاشتراكات والشركات:

```bash
npx prisma db push
npx prisma db seed   # يحدّث حساب الأدمن الأول إلى isSuperAdmin: true
```

حساب `admin@bhd-om.com` يصبح بعد الـ seed بأتم الصلاحيات (isSuperAdmin).
