# قاعدة البيانات — ما هو مربوط في المشروع؟ (مرجع دائم)

**الغرض:** توثيق **الواقع في الكود** حتى لا يُخلط بين «ما يُذكر في README القديم» و«ما يعمل فعلياً»، ومسار **الترقية** من بيئة مؤقتة إلى إنتاج مستقر.

**آخر مراجعة للكود:** 2026-04-02

---

## 1) ماذا يستخدم المشروع فعلياً؟

| البند | القيمة في المستودع الحالي |
|--------|---------------------------|
| **محرك قاعدة البيانات** | **PostgreSQL** فقط |
| **ORM** | **Prisma 7** (`prisma/schema.prisma` → `provider = "postgresql"`) |
| **اتصال التطبيق (Next.js / API)** | `@prisma/adapter-pg` + حزمة **`pg`** — ملف: `lib/prisma.ts` |
| **عنوان الاتصال** | يُقرأ من متغيرات البيئة عبر `lib/env/databaseUrl.ts` |

**مهم:** وجود حزم `better-sqlite3` و`@prisma/adapter-better-sqlite3` في `package.json` **لا يعني** أن التطبيق يستخدم SQLite في التشغيل الحالي — **لا يوجد** في الكود (`lib/prisma.ts`) ربط بـ SQLite؛ الربط الفعلي هو **PostgreSQL** فقط.

## 2) من أين يأتي رابط الاتصال؟ (ترتيب الأسبقية)

الدالة `getDatabaseUrlForRuntime()` تجرب بالترتيب أول قيمة صالحة تبدأ بـ `postgresql://` أو `postgres://`:

1. `DATABASE_URL`
2. `POSTGRES_PRISMA_URL`
3. `POSTGRES_URL`
4. `NEON_DATABASE_URL`
5. `PRISMA_DATABASE_URL`

ثم تُطبَّق معالجات اختيارية (SSL لغير المحلي، مهلة اتصال، `pgbouncer` لمضيف Neon pooler عند الحاجة) — راجع `lib/env/databaseUrl.ts`.

**الإنتاج (مثلاً Vercel):** إذا لم يُعرَّف أي رابط صالح، لا يُنشأ اتصال — تظهر أخطاء مثل «قاعدة البيانات غير متصلة» في الواجهات التي تعتمد على الخادم.

**التطوير المحلي:** إذا لم يُضبط متغير، يُستخدم افتراضياً (غير إنتاج):

`postgresql://postgres:postgres@127.0.0.1:5432/bhd_om`

(يجب أن يكون لديكم PostgreSQL محلي يعمل بهذا الاسم أو تعديلون الرابط في `.env`.)

## 3) الملفات المرجعية في المستودع

| الملف | الدور |
|--------|--------|
| `prisma/schema.prisma` | تعريف الجداول — **PostgreSQL** |
| `prisma.config.ts` | إعداد Prisma CLI (هجرات، seed) — يستخدم نفس حل عنوان الاتصال |
| `lib/prisma.ts` | إنشاء `PrismaClient` للتطبيق |
| `lib/env/databaseUrl.ts` | جمع متغيرات البيئة ومعالجة الرابط |
| `.env.example` | مثال المتغيرات (بدون أسرار) |

## 4) ما المقصود بـ «قاعدة مؤقتة» ثم «ترقية»؟

- **مرحلة مؤقتة (شائعة في البداية):** أي **PostgreSQL** متاح للتطوير أو تجربة الإنتاج (Neon مجاني، Vercel Postgres، سيرفر صغير، إلخ) — طالما الرابط صحيح في **بيئة النشر** والهجرات مُطبَّقة.
- **الترقية (لاحقاً):** الانتقال إلى **خطة أقوى** أو **مزوداً مؤسسياً** (RDS، Azure، إلخ) مع:
  - **نسخ احتياطي** قبل التبديل
  - تحديث **`DATABASE_URL`** (أو المتغير المستخدم) في Vercel
  - إعادة نشر التطبيق
  - تشغيل **`npx prisma migrate deploy`** على القاعدة الجديدة بعد استعادة البيانات إن لزم

لا حاجة لتغيير **نوع** القاعدة من PostgreSQL إلى محرك آخر لأجل «الترقية» — المشروع مبني على PostgreSQL.

للتوسع والمعايير طويلة المدى راجع أيضاً: [SCALING-DATABASE.md](./SCALING-DATABASE.md).

## 5) أوامر شائعة

```bash
# توليد العميل بعد تغيير المخطط
npx prisma generate

# تطبيق الهجرات على قاعدة مربوطة بـ DATABASE_URL في .env
npx prisma migrate deploy

# (تطوير فقط) دفع المخطط بدون ملفات هجرة تفصيلية — بحذر على الإنتاج
npx prisma db push
```

---

## 6) خلاصة لغير المختصين

- الموقع **لا يخزّن** جداول الإنتاج في ملف SQLite داخل المستودع — المصدر الرسمي للبيانات الحرجة على الخادم هو **PostgreSQL** عبر **Prisma**.
- «ربط القاعدة» = وضع **رابط `postgresql://...`** في متغيرات بيئة **الإنتاج** (مثل Vercel) ثم إعادة النشر.
- **GitHub `postgres/postgres`** هو **مشروع برمجيات** لمحرك PostgreSQL، وليس عنوان اتصال لموقعكم.
