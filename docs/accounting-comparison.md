# مقارنة نظام المحاسبة: الموقع السابق (ain-oman-web) والموقع الحالي (bhd-om)

## الموقع السابق (ain-oman-web)

### المصادر
- **دليل الحسابات:** ثابت في الكود (`src/lib/chart-of-accounts.ts`) — مصفوفة JavaScript، لا قاعدة بيانات.
- **الملخص المالي:** من API واحد `/api/financial/summary` يقرأ من **ملفات JSON** في `.data/` (invoices, payments, bookings, contracts, checks, maintenance).
- **لا تحقق هوية** على هذا الـ API — الطلب يُنفَّذ دائماً ويُرجع 200 مع أرقام محسوبة من الملفات.

### الهيكلة
- لوحة مالية واحدة تجلب الملخص من API ثم تعرضه.
- صفحات فرعية (دليل الحسابات، ميزان المراجعة) تعتمد على الدليل الثابت في الكود.
- ميزان المراجعة يعرض أرصدة من `CHART_OF_ACCOUNTS` (كلها أصفار بعد تصفير البيانات الوهمية).

### نقاط القوة
- بسيط وواضح.
- لا يعتمد على الجلسة في الـ API، فلا مشكلة 401.
- يعمل فوراً ما دامت الملفات موجودة.

### نقاط الضعف
- تخزين ملفات (.data/*.json) غير مناسب للإنتاج ولا يتوسع.
- لا قيد مزدوج ولا مستندات/قيود حقيقية.
- دليل الحسابات ثابت ولا يُحدَّث من واجهة.
- لا ربط تلقائي مع دفعات الاشتراك أو الحجز من التطبيق.

## الموقع الحالي (bhd-om)

### المصادر
- **قاعدة بيانات (Prisma/PostgreSQL):** جداول `AccountingAccount`, `AccountingDocument`, `AccountingJournalEntry`, `AccountingJournalLine`, `AccountingFiscalPeriod`.
- **نقطة نهاية موحدة:** `GET /api/accounting/data` تُرجع الحسابات + المستندات + القيود + الفترات بعد مزامنة الحجوزات المدفوعة.
- **تحقق هوية:** الاعتماد على الجلسة (getToken، كوكي الطلب، next/headers، getServerSession) — إن فشل كلها يُرجع 401.

### الهيكلة
- طبقة محاسبة في `lib/accounting/` (dbService، rbac، permissions).
- إيصالات الاشتراك تُسجَّل عند الدفع في `change-with-payment`.
- إيصالات الحجز تُنشأ في `POST /api/bookings` و`syncPaidBookingsToAccountingDb`.
- لوحة الحسابات تجلب من `/api/accounting/data`؛ إن حصل 401 تنتقل للبيانات المحلية (localStorage) الفارغة.

### نقاط القوة
- قيد مزدوج، مستندات، قيود، فترات مالية، تحديث أرصدة تلقائي.
- ربط تلقائي مع الاشتراكات والحجوزات.
- مناسب للتوسع والإنتاج.
- دليل حسابات في قاعدة البيانات مع إمكانية إعادة حساب الأرصدة.

### نقاط الضعف السابقة
- عند فشل قراءة الجلسة في الـ API (مثلاً في بيئة Vercel) كان يُرجع 401 فتعرض الواجهة بيانات محلية فارغة ولا يظهر أي مبالغ.

---

## التعديلات المطبقة (بعد المراجعة)

1. **تعزيز التحقق من الجلسة** (`lib/accounting/rbac/apiAuth.ts`):
   - استخدام عدة أسماء للكوكي وقراءة من `request.cookies` ثم `cookies()` من next/headers.
   - إضافة **getServerSession(authOptions)** كاحتياطي أخير (مشابه لطريقة التحقق في مسارات أخرى في bhd-om).
   - إن وُجدت جلسة بدون دور محدد، اعتبار المستخدم كـ ACCOUNTANT للسماح بعرض الحسابات.

2. **تفضيل API في الإنتاج** في لوحة الحسابات حتى مع عدم تعيين `NEXT_PUBLIC_ACCOUNTING_USE_DB`.

النتيجة المتوقعة: إن كانت الجلسة تعمل عبر getServerSession في بيئة الإنتاج، فستُقبل طلبات `/api/accounting/data` وتظهر المبالغ. إن استمرت المشكلة، السبب غالباً إما عدم وصول الجلسة في Route Handlers (قيود Next.js/next-auth) أو أن البيانات الفعلية غير موجودة في نفس قاعدة البيانات المتصلة من Vercel.

## الخلاصة

| المعيار           | الموقع السابق (ain-oman-web) | الموقع الحالي (bhd-om)        |
|------------------|------------------------------|--------------------------------|
| مصدر البيانات    | ملفات JSON                   | PostgreSQL (Prisma)            |
| قيد مزدوج        | لا                           | نعم                            |
| ربط الاشتراك/الحجز | لا                         | نعم (تلقائي)                   |
| تحقق هوية على API | لا                          | نعم (مع احتياطيات متعددة)     |
| قابلية التوسع    | ضعيفة                        | عالية                          |
| عرض المبالغ      | يعمل إن وُجدت الملفات       | يعمل إن نجح التحقق ووُجدت البيانات |

النظام الحالي **أقوى وهيكلة أفضل**؛ المشكلة كانت في **تحقق الجلسة في بيئة الإنتاج** وتم تخفيفها باحتياطيات إضافية (بما فيها getServerSession).
