# اقرأني — الدليل التقني الشامل لمشروع BHD-OM

**الغرض:** أن يكون هذا الملف **نقطة الدخول الفكرية** لفهم الموقع كمنصة كبيرة قابلة للصيانة: ماذا يفعل كل جزء، أين يُربط، كيف يُصان، وما المخاطر المعروفة. الهدف التشغيلي: **ملايين السجلات ومستخدمين كثر** دون هبوط في الجودة — التوثيق جزء من الهندسة، لا زينة.

**آخر تحديث مبدئي للهيكل:** 2026-03-29 — يُحدَّث عند تغييرات معمارية كبيرة (انظر [من يحدّث هذا الملف؟](#من-يحدّث-هذا-الملف)).

## 3. طبقات النظام (كيف يُفكّك العطل؟)

```
المتصفح (React / Next.js App Router)
    ↓ fetch / router
API Routes (app/api/**) — تحقق جلسة، صلاحيات
    ↓
Prisma → PostgreSQL (مصدر الحقيقة للبيانات الحرجة على الخادم)
    ‖
localStorage (نموذج قديم/مزدوج لبعض الوحدات — انظر القسم 10)
```

- **عند خلل «يظهر للمستخدم X ولا يظهر للأدمن Y»:** غالباً تضارب **مزامنة** بين `GET` من الخادم و`localStorage` — راجع دفتر العناوين و`mergeAddressBookApiWithLocal`.
- **عند خلل مالي/حجز:** ابدأ من **Prisma** (`BookingStorage`, الحجز في API) ثم المحاسبة في `dbService`.

## 4. لوحة الإدارة — صفحات حرجة وربطها

### 4.1 إدارة البيانات `/[locale]/admin/data`

| البند | التفاصيل |
|-------|----------|
| **الفائدة** | تصفير منضبط لقاعدة البيانات (مع الإبقاء على **العقارات** كصفوف)، نسخ احتياطي كامل JSON، استعادة؛ ومنفصل: تصفير **تشغيلي للمتصفح** (localStorage). |
| **الربط في الكود** | `app/[locale]/admin/data/page.tsx`؛ API: `POST /api/admin/data/reset`, `POST /api/admin/data/backup`, `POST /api/admin/data/restore`. |
| **الحماية** | جلسة **ADMIN** + متغير بيئة **`ADMIN_DATA_RESET_PIN`** (8+ أحرف). بدون المتغير تُعاد 503. |
| **منطق التصفير الخادم** | `lib/server/dataResetKeepProperties.ts` (`executeResetKeepProperties`) — يُستدعى أيضاً من `prisma/reset-keep-properties.ts`. |
| **النسخة الاحتياطية** | `lib/server/dataBackupSnapshot.ts` — تصدير/استيراد لقطات `version: 2`. الاستعادة **تمسح القاعدة** ثم تعيد الإدراج — خطرة جداً. |
| **الصيانة** | عند إضافة جدول Prisma جديد: حدّث **التصدير والاستيراد والمسح** في `dataBackupSnapshot.ts` و`executeWipeAllForRestore`. اختبر على نسخة تطوير. |
| **مخاطر** | استعادة ملف قديم قد تكسر FK أو أحجام الطلبات على Vercel؛ التصفير يحذف المستخدمين — الجلسة الحالية تصبح غير صالحة (يُفضّل `signOut`). |

### 4.2 دفتر العناوين `/[locale]/admin/address-book` مقابل «حسابي» `/[locale]/admin/my-account`

| البند | التفاصيل |
|-------|----------|
| **مصدر «حسابي»** | `GET/PATCH /api/user/linked-contact` — صف واحد مرتبط بـ `linkedUserId` في `AddressBookContact`. |
| **قائمة الإدارة** | `GET /api/address-book` — كل الجهات؛ دمج مع محلي عبر `mergeAddressBookApiWithLocal` في `lib/data/addressBook.ts`. |
| **الربط** | الحفظ من «حسابي» يستدعي `syncContactToAddressBookApi` (`POST /api/address-book`). التكرار يُزال على الخادم (`lib/server/addressBookDedupe.ts`) وعلى العميل (`addressBookDedupeShared.ts`). |
| **الصيانة** | أي تغيير في شكل `Contact` يجب أن يبقى متوافقاً مع JSON في Prisma ومع `linked-contact`. |

### 4.3 النسخ الاحتياطي القديم `/[locale]/admin/backup`

| البند | التفاصيل |
|-------|----------|
| **الفائدة** | قد تبقى واجهة قديمة لنسخ **محلي** — راجع `app/[locale]/admin/backup/page.tsx` إن وُجدت. الصفحة **`/admin/data`** هي المرجع الموحّد للخادم + المتصفح حالياً. |

*(إن اختلفت الصفحتان لاحقاً، حدّث هذا القسم و`SITE-SCENARIOS-AND-LINKS.md`.)*

### 4.4 المحاسبة، الحجوزات، الاشتراكات

- **السيناريوهات والروابط:** `docs/SITE-SCENARIOS-AND-LINKS.md` (قسم الحجز، الاشتراك، المحاسبة).
- **الصيانة:** أي API جديد يغيّر مبالغ أو حالات حجز يجب أن يمرّ على نفس قواعد الإيصالات والمراجع في `docs/ACCOUNTING_*` و`dbService`.

## 10. ديون وعيوب معروفة ومراقبتها

| الموضوع | الوصف | إجراء الصيانة |
|---------|--------|----------------|
| **مزدوجية localStorage + PostgreSQL** | جزء من البيانات لا يزال يُحدَّث في المتصفح (حجوزات/عقود قديمة) بينما المصدر الحقيقي يتحول للخادم. | عند لمس منطق قديم: تفضيل **API + Prisma** وتقليل الاعتماد على المحلي تدريجياً. |
| **حجم نسخة JSON الكاملة** | النسخ الاحتياطي من `/admin/data` قد يكون كبيراً — حدود Vercel/الوقت. | لقواعد ضخمة: لاحقاً استخدام `pg_dump` أو تخزين S3 بإشراف DBA. |
| **استعادة كاملة** | تستبدل كل الجداول — خطأ في ملف يكسر الإنتاج. | دائماً نسخة احتياطية قبل الاستعادة؛ اختبار على staging. |
| **تعدد صفوف دفتر العناوين لنفس الهاتف** | يُدار بالدمج والـ dedupe؛ يحتاج اتساق `linkedUserId`. | مراقبة `GET /api/address-book` والمنطق في `addressBookDedupeShared.ts`. |
