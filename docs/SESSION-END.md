# نهاية الجلسة — إجراءات الإغلاق والتحديث
# Session End — Closing & Update Checklist

**الغرض:** في نهاية كل جلسة عمل، تنفيذ هذه الخطوات لضمان رفع كل التغييرات، تحديث الملفات المرجعية، وإصلاح أي تقصير — مع توثيق التاريخ والوقت. يضمن العمل من أكثر من جهاز عدم فقدان السياق أو تناقص البرمجة.

**آخر تحديث لهذا الملف:** 2026-03-03

---

## 1. أوامر نهاية الجلسة (للمساعد / AI)

عند انتهاء الجلسة (قبل إغلاق المحادثة أو الانتقال لجهاز آخر):

1. **تحديث ملف السياق اليومي** `docs/DAILY-CONTEXT.md`:
   - إضافة تاريخ ووقت نهاية الجلسة.
   - تلخيص ما تم في المحادثة (مهام، قرارات، ملفات مُعدّلة).
   - تسجيل الأحداث الجديدة (ميزات، إصلاحات، تغييرات مهمة).
   - الإبقاء على كل شيء مُؤرّخ ومُؤقّت.

2. **مراجعة التغييرات**:
   - التأكد من أن كل الملفات المُعدّلة مكتملة ولا تحتوي أخطاء (lint/build إن أمكن).
   - إن وُجد تقصير أو شوائب (نص ناقص، TODO مهم، سلوك غير مكتمل) — إصلاحها قبل الإغلاق.

3. **تحديث الملفات المرجعية** إن لزم:
   - إن أُضيفت صفحة أو مسار جديد: تحديث `docs/SITE-SCENARIOS-AND-LINKS.md`.
   - إن تغيّر سير عمل أو صلاحيات: تحديث الوثائق ذات الصلة (`BOOKING_WORKFLOW_DESIGN.md`, `ROLES-SUBSCRIPTION-DESIGN.md`, إلخ).

4. **رفع كل التغييرات إلى المستودع — دائماً في نهاية الجلسة**:
   - `git add` لجميع الملفات المُعدّلة (أو `git add -A`) — بما فيها `docs/DAILY-CONTEXT.md`, `docs/SESSION-START.md`, `docs/SESSION-END.md` إن تغيّرت.
   - `git commit` مع رسالة واضحة ومُؤرّخة (يمكن تضمين التاريخ في الرسالة).
   - `git push origin master` (أو الفرع المعتمد) — **لا تُغلق الجلسة دون الرفع** لضمان عدم فقدان العمل عند التنقّل بين الأجهزة.

5. **التأكد من النجاح**:
   - إن كان المشروع يُبنى على Vercel أو CI، التأكد من أن البناء ينجح بعد الرفع (أو تنبيه المستخدم بذلك).

## 2. قالب تحديث `DAILY-CONTEXT.md` في نهاية الجلسة

أضف قسماً جديداً في أعلى الملف (تحت "آخر الأحداث") بالشكل التالي:

```markdown
### جلسة [تاريخ] — [وقت البداية]–[وقت النهاية] (UTC أو التوقيت المحلي)

- **ما تم:** [ملخص المهام: مثلاً إصلاح بناء، إضافة صفحة X، تحديث وثيقة Y]
- **الملفات المُعدّلة:** [قائمة الملفات الرئيسية]
- **أحداث جديدة:** [ميزات، إصلاحات، قرارات تقنية]
- **ملاحظات للجلسة القادمة:** [إن وُجدت]
```

---

## 3. قائمة تحقق سريعة (Checklist)

- [ ] تم تحديث `docs/DAILY-CONTEXT.md` بتاريخ ووقت وملخص الجلسة.
- [ ] تمت مراجعة الكود والملفات — لا أخطاء واضحة أو تقصير مهم.
- [ ] تم تحديث وثائق المشروع إن لزم (SITE-SCENARIOS، WORKFLOW، إلخ).
- [ ] تم رفع كل التغييرات: `git add -A`, `git commit`, `git push origin master`.
- [ ] البناء ينجح (أو تم التنبيه بالمشاكل).

---

## 4. ملاحظات للعمل من أكثر من جهاز

- **في بداية الجلسة — دائماً اسحب:** على أي جهاز، نفّذ `git pull origin master` قبل البدء.
- **في نهاية الجلسة — دائماً ارفع:** نفّذ `git push origin master` قبل إغلاق الجلسة؛ لا تُغلق دون الرفع.
- **اقرأ `docs/SESSION-START.md`** و **`docs/DAILY-CONTEXT.md`** في بداية الجلسة على الجهاز الجديد لاستعادة السياق.
- **لا تعتمد على ذاكرة المحادثة فقط** — كل ما يُسجّل في `DAILY-CONTEXT.md` يبقى في المستودع ويُحدَّث مع كل جلسة.
