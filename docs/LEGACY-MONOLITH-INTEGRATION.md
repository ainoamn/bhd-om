## 1. ملخص تنفيذي

تم اتخاذ قرار تغيير التكتيك بالكامل:

| قبل | بعد |
|-----|-----|
| تطبيق مكتبي (Electron + SQLite) | **موقع ويب لايف** على الإنترنت |
| ملف HTML واحد ضخم كمنتج نهائي | **Next.js + PostgreSQL** كمنتج نهائي |
| تخزين محلي (`localStorage` / SQLite) | **PostgreSQL** مصدر الحقيقة (server-first) |
| نافذة Electron منفصلة | **نفس الموقع** — مع إمكانية «تثبيت التطبيق» (PWA) لاحقاً |

**لماذا لم نستبدل الموقع الحالي؟**  
موقع `bhd-om` (Next.js) منشور على [bhd-om.com](https://bhd-om.com)، مربوط بـ PostgreSQL، ويحتوي موقعاً عاماً + لوحة إدارة + محاسبة + حجوزات. النظام المحلي (`عقود الايجار`) أغنى في **التشغيل الداخلي** (عقود الوحدات، حجوزات المباني، طباعة A4، مهام…) لكنه غير مناسب كنشر مباشر على الويب.

**الحل:** رفع النظام المحلي إلى Git تحت `legacy/` كمرجع، ثم **نقل المنطق تدريجياً** إلى جذر `bhd-om` — دمج ما نريد، وتعطيل ما لا نحتاجه.

## 2. ما هو كل مشروع؟

### 2.1 جذر المستودع — `bhd-om` (المنتج الرئيسي)

- **الغرض:** موقع الشركة العام + لوحة إدارة + عمل لايف على الإنترنت.
- **التقنية:** Next.js 16, React 19, TypeScript, Prisma, PostgreSQL, next-intl.
- **النشر:** Vercel + Neon/Vercel Postgres (انظر `DEPLOYMENT.md`).
- **المستخدم يرى:** نفس الواجهة دائماً — لا «وضع مكتبي» ولا «وضع سحابة».

### 2.2 المجلد `legacy/bhd-real-estate/` (المرجع — لا يُنشر كموقع مستقل)

نسخة مصدرية من المشروع المحلي `C:\dev\عقود الايجار`، رُفعت في commit `17a67ff` (يونيو 2026).

| المسار | الوصف |
|--------|--------|
| `bhd-real-estate.html` | SPA monolith — الواجهة + كل المنطق (~68k سطر) |
| `scripts/bhd/*.js` | نفس المنطق مقسّماً (01-core … 06-maintenance-init) |
| `desktop/` | Electron + SQLite — **يُستغنى عنه** في الاستراتيجية الجديدة |
| `apps/api/` | Express API + PostgreSQL (مسار سحابة منفصل — **يُوحَّد** لاحقاً مع Prisma في الجذر) |
| `server/` | خادم محلي يعرض HTML على المنفذ 3789 |
| `deploy/` | Docker Compose للإنتاج (مسار VPS بديل) |
| `docs/` | خطط السحابة، الهجرة، الاختبار |

**ما لم يُرفع إلى Git (محلي فقط):**

- `node_modules/`
- `.env` وملفات الأسرار
- `backups/` و `data/`
- قواعد بيانات `*.db` (مثل `rental.db`)

## 3. لماذا نُقل المشروع إلى Git؟

1. **حفظ الاستثمار:** سنوات من المنطق (عقود، محاسبة، حجوزات، طباعة، صلاحيات) لا تُفقد عند الدمج.
2. **عمل من أكثر من جهاز:** المستودع على GitHub؛ المجلد المحلي للعمل: `C:\dev\bhd-om`.
3. **مرجع للمطورين و AI:** عند نقل ميزة من monolith إلى Next.js، يُقرأ الكود من `legacy/` دون البحث على القرص فقط.
4. **عدم كسر الموقع الحي:** وضع الكود تحت `legacy/` يبقي `bhd-om.com` يعمل دون تغيير.
5. **خارطة دمج واضحة:** مقارنة شاشة بشاشة بين `/admin/*` في Next.js و monolith.

## 5. ما تم إنجازه في جلسات تطوير النظام المحلي (قبل النقل)

هذه التحسينات وُجدت في `legacy/bhd-real-estate` عند الرفع (يونيو 2026):

### 5.1 أداء زر «تحديث النظام» والإقلاع المكتبي

**المشكلة:** الضغط على «تحديث النظام» أو فتح التطبيق المكتبي كان يستغرق نحو دقيقة — تجميد الواجهة.

**السبب:**

- سحب كامل لكل البيانات من SQLite عبر IPC ثم نسخها إلى `localStorage` و IndexedDB **ثلاث مرات**
- بعد إعادة التحميل: مزامنة سحابة، مزامنة محاسبة كاملة لكل العقود، إصلاح مرفقات… كلها **متزامنة** وتعطل الواجهة

**الإصلاح (في `scripts/bhd/02-data-store.js` و `06-maintenance-init.js` و `bhd-real-estate.html`):**

- مسار سريع `bhd_app_refresh_fast` — تخطي إعادة قراءة القاعدة بعد التحديث
- إلغاء السحب المكرر قبل `location.reload()`
- تأجيل مزامنة السحابة والمحاسبة للخلفية
- على المكتبي: تخطي `kvGetBulk` إذا كانت البيانات موجودة في `localStorage`
- حفظ **المفاتيح المتغيرة فقط** (dirty tracking) بدل كل الـ KV
- إلغاء نسخ IndexedDB المتزامن على المكتبي؛ تأجيله لـ `requestIdleCallback`
- مهلة 2.5 ثانية لفحص API السحابة بدل انتظار طويل

> **ملاحظة:** هذه الإصلاحات تخص المسار المكتبي القديم. عند الدمج في Next.js لا نعيد نفس أخطاء المعمارية.

### 5.2 إصلاحات أداء سابقة (سجل العمل)

موثّقة في `legacy/bhd-real-estate/SESSION_WORK_LOG.md`:

- إزالة `syncAllAccountingFromSavedContracts()` من المهام التلقائية الثقيلة
- تخزين مؤقت لخريطة العقود وتقليل تكرار `loadDashboardAux`
- إصلاح O(n²) في المحاسبة (`getManualEntryCoveredChequeIds`)
- مزامنة تلقائية في الخلفية عبر `requestIdleCallback`

## 6. مقارنة سريعة: ماذا نأخذ من أين؟

| القدرة | `bhd-om` (الجذر) | `legacy` (monolith) | القرار |
|--------|------------------|---------------------|--------|
| موقع عام (عقارات، مشاريع، SEO) | ✅ جاهز | ❌ | **يبقى في الجذر** |
| تسجيل دخول + أدوار | ✅ Prisma + NextAuth | محلي | **يبقى في الجذر** |
| محاسبة | ✅ جداول + واجهة admin | ✅ أثقل في monolith | **دمج تدريجي** — مقارنة قبل النقل |
| دفتر العناوين | ✅ server-first | ✅ `?mode=addressbook` | **موحّد** — PostgreSQL مصدر الحقيقة؛ legacy للاستخدام اليومي + استيراد للعقود |

### 6.1 ربط دفتر العناوين (موقع ↔ legacy)

| الواجهة | الرابط | الدور |
|---------|--------|--------|
| إدارة كاملة (CRUD) | `/ar/admin/address-book` | إضافة/تعديل/أرشفة — مصدر الحقيقة في PostgreSQL |
| استخدام تشغيلي | `/api/admin/legacy-real-estate/bhd-real-estate.html?mode=addressbook` | عرض متزامن + استيراد للعقود + اتصال/واتساب/باركود/طباعة |

**آلية المزامنة (موقع → legacy):**

1. عند فتح HTML القديم يُحقَن سكربت `bhd-site-bridge-boot` مع بيانات الجلسة.
2. `GET /api/admin/legacy-bridge/bootstrap` يجلب حتى 500 مستخدم + كل جهات الاتصال غير المؤرشفة من Prisma.
3. `mapSiteContactToLegacyEntry()` يحوّل كل `AddressBookContact` إلى صيغة `bhd_address_book` في `localStorage` (مع `siteContactId`, `userId`, `serialNumber`, `extraMobile`).
4. بعد التطبيق يُعاد تحميل `addressBookEntries` ويُحدَّث الجدول تلقائياً في وضع `addressbook`.

**ميزات من الموقع في legacy:**

| الميزة | السلوك |
|--------|--------|
| 📞 اتصال | `tel:` على رقم الجوال |
| 💬 واتساب | `https://wa.me/968…` (نفس منطق الموقع) |
| ✉️ بريد | `mailto:` |
| 🖨️ طباعة | `GET /api/admin/legacy-bridge/address-book/print?id={siteContactId}` — تقرير A4 مع شعار الشركة وباركود QR إن وُجد مستخدم مربوط |
| ▦ باركود | QR يشير إلى `https://www.bhd-om.com/{ar\|en}/scan/{userId}` |

**ماذا يحدث عند مسح الباركود؟**

1. الهاتف يفتح صفحة عامة: `/{locale}/scan/{userId}` (لا تتطلب تسجيل دخول).
2. الصفحة تستدعي `GET /api/scan/{userId}` وتعرض بطاقة المستخدم: الاسم، الرقم المتسلسل، الهاتف، البريد، الاشتراك، ملخص نشاط.
3. من البطاقة يمكن الاتصال أو فتح واتساب مباشرة.
4. **لا علاقة لمسح الباركود بلوحة Vercel** ([deployments](https://vercel.com/bhdom89-8158s-projects/bhd-om/deployments)) — تلك لوحة نشر للمطورين فقط. الباركود يوجّه دائماً إلى الموقع العام `bhd-om.com`.

**طباعة التقرير:** تُنشأ على الخادم من بيانات PostgreSQL الحالية (ليست نسخة `localStorage` القديمة). عند وجود `userId` مربوط يُضمَّن QR في التقرير المطبوع مثل صفحة `/admin/address-book`.

---
| حجوزات / عقود | ✅ API + صفحات admin | ✅ أغنى (وحدات، تجديد، إلغاء) | **نقل المنطق الناقص** من legacy |
| طباعة A4 موحّدة | جزئي | ✅ معايير قوية | **نقل المعايير** إلى الجذر |
| Desktop / SQLite | ❌ | ✅ | **تعطيل** |
| `apps/api` منفصل | ❌ (Prisma مدمج) | ✅ Express | **لا تشغيل APIين** — توحيد على Prisma |
